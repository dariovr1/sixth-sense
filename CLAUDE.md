# Sixth Sense — Project Instructions for Claude Code

This repository contains the Sixth Sense plugin. If you are working ON this repo (developing the plugin), follow these instructions.

## Versioning note: public 1.0.0 reset (2026-07-17)
`plugin.json` was reset from `1.3.3` to `1.0.0` at first public release. The `1.0.x`-`1.3.3` numbers referenced below (and in git history) are the real internal iteration count from private development; they still describe when each fix actually landed and shouldn't be edited away. Just don't expect them to line up with the live `plugin.json` version going forward — future bumps count up from `1.0.0`, not from `1.3.3`.

## Stack
- Skills are Markdown files with YAML frontmatter
- Tests are mostly manual (verify skills load correctly), plus `node scripts/preflight.js` for the checks that can be automated (BOM, hook JSON validity, frontmatter, `session-start.js` envelope invariants)
- Use `/context` to check the token weight of the skills

## Conventions
- Every skill lives at `skills/<skill-name>/SKILL.md`
- The `name` field in the frontmatter MUST match the directory name
- Model-invoked skills have a `description` rich in trigger phrases
- User-invoked skills have `disable-model-invocation: true`
- Keep skills under 5,000 tokens to optimize progressive disclosure

## Known Bug: Skill/Command Name Collisions Break Skill Resolution
`claude plugin details sixth-sense@sixth-sense` lists 23 "Skills" even though `skills/` only has 17 directories — the 6 extras were skills that also had a `commands/<name>.md` wrapper of the identical name (`code-review`, `grill-with-docs`, `implement`, `setup-sixth-sense`, `tdd`, `to-spec`). This was first assumed to be a harmless display artifact. **It is not.** Verified against two real headless-run session transcripts (2026-07-11): invoking `Skill(skill: "sixth-sense:code-review")` resolved to the **command**'s body (`commands/code-review.md`, no "Base directory for this skill" header), not the skill's. Since that command's own body then told the model to invoke `Skill(skill: "code-review")` (bare, no plugin prefix) to reach "the code-review skill", and Claude Code ships its own unrelated built-in skill also named `code-review`, that second call silently resolved to the **wrong, foreign skill** in both runs — sixth-sense's actual code-review skill (git-repo fallback, Dylan Dog premortem/hunting linkage) never executed in either benchmark.

**Fixed for all 6.** Every skill that shared its name with a `commands/<name>.md` wrapper was renamed (directory + frontmatter `name`), collision-free from its sibling command and from anything external:

| Command (unchanged) | Old skill name | New skill name |
|---|---|---|
| `/sixth-sense:code-review` | `code-review` | `requesting-code-review` |
| `/sixth-sense:tdd` | `tdd` | `test-driven-development` |
| `/sixth-sense:implement` | `implement` | `implementing-work` |
| `/sixth-sense:to-spec` | `to-spec` | `synthesizing-spec` |
| `/sixth-sense:setup-sixth-sense` | `setup-sixth-sense` | `configuring-sixth-sense` |

`grill-with-docs`/`grilling-with-docs` is no longer in this table — it was removed entirely (2026-07-11), not just renamed: it was a one-line composition (`grilling` + `domain-modeling`, no logic of its own) that duplicated the picker-clutter problem this same session had just fixed for the tdd-stack skills. Its persistence behavior is now a conditional inside `grilling` itself (see that skill's "Persisting Decisions" section) instead of a separate command+skill pair.

`code-review` was the only one *confirmed* broken (external collision with a Claude Code built-in skill of the identical bare name, verified against two headless-run transcripts, 2026-07-11). A third run (also 2026-07-11) additionally showed `tdd` silently short-circuiting: the model got the command's one-line body and never bothered completing the second hop to load the real skill at all — no collision, just the full discipline (Iron Law, Confirm Interfaces gate, minimal-implementation ladder) never actually loading. The other 4 were not confirmed broken by direct observation, but share the identical structure, so they got the same fix preventively rather than waiting for a transcript to prove it.

Each command's body now tells the model to invoke the fully-qualified `sixth-sense:<new-name>` explicitly, and every cross-reference elsewhere in the plugin (`using-sixth-sense`'s auto-trigger table, `crossroad-stack`, `dylan-dog`, `dispatching-parallel-agents`, `to-spec`→`synthesizing-spec`'s own cross-refs, README) was updated to match. If you add a 7th command/skill pair with the same name in the future, give the skill a distinct name from day one — don't wait to find out the hard way.

**This is not a temporary workaround.** Anthropic has closed the equivalent report as "not planned" ([anthropics/claude-code#15842](https://github.com/anthropics/claude-code/issues/15842); see also [#50486](https://github.com/anthropics/claude-code/issues/50486) on automatic skill namespacing, still open/unimplemented). Renaming skills to be collision-free is the durable fix, not a stopgap — don't revert this the next time a command/skill pair looks like it "should" share a name.

## Skill Count: Keep It Near 8-12, Audit Before Adding
Community consensus (multiple sources, 2026) puts the productive range at 8-12 well-chosen skills before "context tax" sets in — every loaded skill costs tokens whether or not it fires. We're at 16 real skill directories (down from 20 pre-cleanup; `handing-off` was added back deliberately after the cleanup, see README). `dispatching-parallel-agents` writes `.sixth-sense/dispatch-log.md` in every project that uses it — periodically check a real project's log (and which model-invoked skills actually triggered) before adding a new skill. If something hasn't fired in weeks of real use, that's a stronger signal to remove or fold it than any amount of "might be useful" reasoning. Don't chase scale for its own sake. `obra/superpowers` itself only has 14 skills and no agent-tier system at all (verified via the GitHub API on 2026-07-17); we're already past it at 16. The earlier version of this note claimed superpowers had "232 skills, 60 agents" — that number was never checked against the actual repo and turned out to be wrong. Growth for its own sake is the opposite of why this plugin forked away from it, whatever the real count is.

## Removed: `writing-great-skills` (2026-07-11)
Was a real, substantial skill (not a thin wrapper like `grilling-with-docs`) — but topically off-mission: it's about authoring Claude Code skills in general, not about disciplined software development, which is this plugin's stated scope. Removed on request rather than kept for "might be useful someday."

## Considered, Not Adopted (2026-07-11)
Evaluated against `mattpocock/skills` v1.1 and `AgriciDaniel/claude-obsidian`, rejected with reasons — recorded so these don't get re-proposed without re-litigating:
- **`wayfinder`-style GitHub-issue "map" for foggy, multi-session planning.** Genuinely different problem than what's here: it needs a GitHub issue tracker present, and this plugin runs plenty of git-less/scratch sessions where that's not available. `dispatching-parallel-agents`'s `TaskCreate`/`TaskUpdate` dependency graph (`addBlockedBy`) already covers the "break large work into ordered, trackable units" need without an external tracker dependency — a lighter-weight functional analog, not a gap.
- **Moving refactor out of the TDD loop into code review** (Pocock's v1.1 rationale: keeps implementation sessions focused). Rejected: red-green-**refactor** per cycle is core TDD discipline, not overhead — deferring all refactoring to review lets cruft accumulate through an entire implementation with nothing pushing back until `requesting-code-review` runs. Dylan Dog/`opus-review` are for catching what per-cycle refactor structurally can't see, not a replacement for it.
- **A second-brain / PKM layer** (`claude-obsidian`'s ingest/wiki/query model). Solves a different problem — general knowledge management across an Obsidian vault — than this plugin's stated scope of disciplined software development. Requires its own vault and 15 skills of its own; merging it in would double the skill count for a capability outside the mission, exactly the kind of scope creep the 8-12 guideline above exists to prevent. `CONTEXT.md`/ADRs (`domain-modeling`) and `handing-off` already cover this plugin's actual need — not re-explaining a specific project's decisions — without becoming a general note-taking tool.

## Benchmark Fix & Known Regressions (2026-07-20)
Surfaced by an independent claude.ai session that cloned this repo and actually ran `scripts/benchmark.js`, shared with the maintainer and re-verified against the real files in a separate session before anything was changed.

**`scripts/benchmark.js` was wrong, and wrong in the direction that flattered the README.** It compared the removed `writing-great-skills` (see above) against Superpowers' `writing-skills` and counted the missing files as "100% saved" instead of not comparing them at all, and it referenced `agents/code-reviewer.md`, which never existed — the real files are `agents/dylan-dog-triage.md`/`agents/dylan-dog-hunter.md` — so Dylan Dog, the most advertised feature, contributed zero tokens to the reported savings. Fixed: the phantom comparison is gone, Dylan Dog's real weight (~3,540 tokens) is now its own labeled line instead of folded into an opaque "additional files" bucket. **Corrected net savings: 25% at the time of this fix (re-measured 2026-07-21: 23%, Superpowers has since changed slightly), not the 54% the broken script printed.** The README's 44% is not wrong, but it answers a different question than the benchmark script does — it's a static total-footprint comparison (all always-loaded `SKILL.md`/agent-tier text, Superpowers vs. Sixth Sense), not an itemized net-of-added-capability comparison like `scripts/benchmark.js` produces. Revisited deliberately on 2026-07-21, not silently: README now states both numbers side by side (footprint 44%, itemized net savings 23%) with an explanation of what each one actually measures, instead of picking one and hiding the other.

**Two regressions the static comparison has always had, never declared as trade-offs before now:** `using-sixth-sense` (injected into every session unconditionally via the hook) is about 83% *heavier* than Superpowers' `using-superpowers` (1,472 vs 803 tokens, re-measured 2026-07-21 via `node scripts/benchmark.js` — was recorded as 1,458/82% at the time this note was first written, grown slightly since) — the auto-trigger table, `<SUBAGENT-STOP>` guard, and handoff-suggestion logic cost real, unconditional tokens every session, unlike the rest of the plugin where progressive disclosure means an unfired skill's body never loads. `dispatching-parallel-agents` is 18% heavier (2,061 vs 1,742) — the effort-routing table and batching/cap logic cost more than they save relative to Superpowers' simpler dispatch skill. Both are accepted trade-offs (the capabilities they buy — stack-aware auto-triggering, cost-tiered dispatch — don't exist in Superpowers at all), but they were absent from this file's usual practice of naming every known compromise explicitly.

## Dogfooding Discipline (2026-07-13)
Two gaps surfaced from a real project session (an `obsidian` vault using sixth-sense for its own backlog tooling), not from a planned audit — see `agents/sonnet-research.md` and the `<SUBAGENT-STOP>` rewrite in `using-sixth-sense/SKILL.md`, both landed in 1.3.3 from the same session.

1. **Eat your own dispatch-economy dogfood.** This plugin tells every other project to route mechanical work to `haiku-batch`/`sonnet-low` via `dispatching-parallel-agents`, but nothing forced that discipline onto changes to *this* repo — small, well-defined edits here (a single-file rename, a doc update mirrored across README/spec, a one-line routing-table addition) were still being done by hand in the main session instead of dispatched. When working on sixth-sense itself and a task cleanly fits the Effort Routing table in `dispatching-parallel-agents/SKILL.md`, use it — the same criteria apply to this repo as to any other project.
2. **Real usage is a bug channel, not just a side effect.** The `/dylan-dog` command that was promised but never existed (see the retrospective section above) and both 1.3.3 fixes were found the same way: using the plugin on a real project, not running a planned audit. Planned audits (`dylan-dog`, `code-review`) find what you already know to look for; a real project surfaces what nobody thought to check. When a dogfooding session (this repo's own dev work, or a project like the `obsidian` vault that uses sixth-sense daily) turns up friction or a gap, treat it as a first-class bug report — log it here or fix it directly — instead of letting it stay implicit in a transcript nobody re-reads.

## `using-sixth-sense` model-invocation disabled (2026-07-21)
`skills/using-sixth-sense/SKILL.md` now carries `disable-model-invocation: true`. It didn't before, even though `hooks/session-start.js` already injects this exact file's body (frontmatter stripped) unconditionally at every `SessionStart` — meaning the auto-trigger path (its description, "Use when starting any conversation," matches essentially every first message) was pure redundancy on top of a guarantee the hook already provides deterministically, and a real risk of double-injecting the same content if the model also fired the skill on top of the hook. The hook injection is unaffected by this flag and keeps working exactly as before; explicit invocation by name (`Skill(sixth-sense:using-sixth-sense)`) still works too, which matters for reloading it after a context compaction the hook doesn't re-fire for. On Antigravity/Gemini CLI, which has no equivalent to this hook, the platform's own "skill metadata loaded at session start" mechanism is what's relied on instead — the same situation five other `disable-model-invocation: true` skills in this plugin (`configuring-sixth-sense`, `dylan-dog`, `handing-off`, `implementing-work`, `synthesizing-spec`) are already in, so this isn't a new cross-platform risk.
